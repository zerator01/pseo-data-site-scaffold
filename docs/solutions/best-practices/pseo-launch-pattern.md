# pSEO Launch Pattern

Use this as the baseline launch doctrine for any new data-led pSEO site:

1. Fix data truth before SEO amplification.
2. Separate representative views from full raw rankings.
3. Index in layers, not all at once.
4. Use a manifest-driven freshness system instead of hardcoded update dates.
5. Separate lightweight routing indexes from full detail datasets.
6. If direct imports become too heavy, switch to a lazy-loading data access layer with cached lookup maps.
7. Treat page families as a matrix: hub pages, entity pages, and cross pages may all be first-class surfaces.
8. Build anti-orphan hub structure on purpose. State/category hubs should collect long-tail pages.
9. Treat the home page as a navigation matrix with clear entity, geography, and ranking entry paths.
10. Centralize schema generation once multiple dynamic templates exist.
11. Write a site blueprint before large-scale page generation so URL families and link flow do not drift.
12. Write as a transparent publisher, not a fake authority.
13. Give every major page a next-step comparison path.
14. Prefer workflow docs and pattern docs over a large skill surface.
15. Do not expose aspirational navigation. If a tool, ranking, or comparison page is not real yet, keep it out of core entry points.
16. Do not let blueprint scale drift far beyond actual data coverage. Launch doctrine must reflect what the dataset can support now.
17. Replace default framework README and template governance files early so repo truth matches product truth.

If a new project deviates from one of these, document why.
